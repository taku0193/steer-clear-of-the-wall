# Project Structure

責務別に分けたクライアントサイドゲーム構成を基本にします。Next.jsはアプリの入口とビルド基盤として使い、ゲームの状態管理、カメラ、姿勢検出、Canvas描画はクライアント側の明確な境界内に置きます。

カメラ、姿勢検出、描画、ゲーム進行、判定、スコアは独立して理解できるようにし、ブラウザAPIや外部ライブラリへの依存が純粋なゲームロジックへ広がらないようにします。

## Directory Patterns

### Next.js App Entry

Location: `app/`

Next.js App Routerの入口を置きます。

- `app/layout.tsx`: HTML骨格とグローバルCSS読み込み。
- `app/page.tsx`: ゲームアプリを表示するページ。
- `app/globals.css`: アプリ全体のグローバルスタイル。

ページ層は薄く保ち、ゲーム進行やカメラ処理を直接持たせません。

### Application Shell

Location: `src/`

ルートゲームコンポーネント、画面全体の状態、ページレベルの構成を扱います。

Next.js配下で実行する場合、ブラウザAPIを使うルートコンポーネントはクライアントコンポーネントとして扱います。

### Game Domain

Location: `src/game/`

ゲーム進行、壁パターン、衝突判定、スコア、速度、型定義など、UIフレームワークから独立しやすいロジックを置きます。

例:

- 壁パターンのデータ構造
- 姿勢ランドマークから判定用ポイントを作る処理
- 成功・失敗・スコアの計算
- 壁速度とゲームループの純粋ロジック

### Camera and Pose Adapters

Location: `src/camera/` and `src/pose/`

ブラウザのMediaDevices API、映像ストリーム、姿勢検出ライブラリの初期化を扱います。

外部ライブラリ固有の型やライフサイクルはこの層へ閉じ込めます。サーバーコンポーネントから直接参照しないでください。

### Rendering

Location: `src/rendering/`

Canvas描画、アバター描画、壁描画、エフェクトなどを扱います。

描画に必要な入力データは、ゲーム状態や姿勢情報から変換された明確な型にします。描画層はスコアや判定の決定権を持ちません。

### UI Components

Location: `src/components/`

スタート画面、カメラ許可UI、HUD、結果画面、エラー表示などを置きます。

コンポーネントは状態とイベントを受け取って表示し、純粋なゲームロジックを直接抱え込みすぎないようにします。

## Naming Conventions

- Reactコンポーネント名はPascalCaseにする。
- 通常の関数、変数、モジュール内ヘルパーはcamelCaseにする。
- 型、インターフェース、ゲーム状態の名前は意味が分かるドメイン語彙を使う。
- 壁パターンや姿勢名は、UI表示名と内部IDを分ける。

## Dependency Direction

依存方向は次を基本にします。

- `app/` は `src/` のゲームアプリを表示する。
- UIはゲーム状態とイベントを利用する。
- ゲームロジックはReactやNext.jsに依存しない。
- 判定とスコアはカメラやCanvasに依存しない。
- 姿勢検出アダプターは外部ライブラリに依存してよいが、その型を広範囲に漏らさない。
- 描画層はCanvasなどの描画APIに依存してよいが、スコアや判定の決定権を持たない。

## Import Organization

パスエイリアスを導入する場合は、`@/` をプロジェクトルートまたは `src/` に対応させます。導入しない場合は、近いファイル間では相対importを使い、深い相対パスが増えた時点で設計フェーズで見直します。

## Code Organization Principles

- 画面遷移は、開始前、カメラ準備中、カウントダウン、プレイ中、結果、エラーのように明確な状態として扱う。
- 壁パターンは、描画情報と判定情報を混同しすぎない。
- 衝突判定は、調整値をデータとして持てるようにし、難易度を調整しやすくする。
- カメラや姿勢検出が使えない環境でも、エラー表示やモック姿勢で挙動確認できる構成にする。
- 参考リポジトリのディレクトリやファイル名をそのまま再現しない。

## Documentation Rules

- READMEには、セットアップ、起動方法、カメラ利用条件、確認コマンドを書く。
- `.kiro/specs/` には、機能ごとの要求、設計、タスクを置く。
- `.kiro/steering/` は、実装履歴ではなく長期的な判断基準を書く。
