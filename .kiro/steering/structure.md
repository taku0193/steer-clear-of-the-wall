# Project Structure

このプロジェクトは初期段階です。2026-06-24時点では `README.md` と
プロジェクトメモリが中心で、`package.json` と `src/` はまだありません。
ここでは、今後実装を始めるときの構成方針を定義します。

## Organization Philosophy

責務別に分けたクライアントサイドゲーム構成を基本にします。

カメラ、姿勢検出、描画、ゲーム進行、判定、スコアは独立して理解できるようにし、
ブラウザAPIや外部ライブラリへの依存が純粋なゲームロジックへ広がらないようにします。

## Expected Directory Patterns

実装開始後は、次のような責務分離を目安にします。これは固定のファイル一覧ではなく、
新しいコードを配置するときの判断基準です。

### Application Entry

Location: `src/`

アプリの起点、ルートコンポーネント、グローバルスタイルを置きます。
初期化、画面全体の状態、ページレベルの構成を扱います。

### Game Domain

Location: `src/game/`

ゲーム進行、壁パターン、衝突判定、スコア、レベル、型定義など、
UIフレームワークから独立しやすいロジックを置きます。

例:

- 壁パターンのデータ構造
- 姿勢ランドマークから判定用ポイントを作る処理
- 成功・失敗・スコアの計算

### Camera and Pose Adapters

Location: `src/camera/` または `src/pose/`

ブラウザのMediaDevices API、映像ストリーム、姿勢検出ライブラリの初期化を扱います。
外部ライブラリ固有の型やライフサイクルはこの層へ閉じ込めます。

### Rendering

Location: `src/rendering/` または `src/canvas/`

Canvas描画、アバター描画、壁描画、エフェクトなどを扱います。
描画に必要な入力データは、ゲーム状態や姿勢情報から変換された明確な型にします。

### UI Components

Location: `src/components/`

スタート画面、カメラ許可UI、HUD、結果画面、エラー表示などを置きます。
ゲームロジックを直接持たず、状態とイベントを受け取って表示します。

## Naming Conventions

- ファイル名は、プロジェクト初期化時に採用するフレームワークの慣習に合わせる。
- Reactコンポーネントを使う場合、コンポーネント名はPascalCaseにする。
- 通常の関数、変数、モジュール内ヘルパーはcamelCaseにする。
- 型、インターフェース、ゲーム状態の名前は意味が分かるドメイン語彙を使う。
- 壁パターンや姿勢名は、UI表示名と内部IDを分ける。

## Dependency Direction

依存方向は次を基本にします。

- UIはゲーム状態とイベントを利用する。
- ゲームロジックはUIフレームワークに依存しない。
- 判定とスコアはカメラやCanvasに依存しない。
- 姿勢検出アダプターは外部ライブラリに依存してよいが、その型を広範囲に漏らさない。
- 描画層はCanvasなどの描画APIに依存してよいが、スコアや判定の決定権を持たない。

## Import Organization

パスエイリアスを導入する場合は、`@/` を `src/` に対応させることを候補にします。
導入しない場合は、近いファイル間では相対importを使い、深い相対パスが増えた時点で
設計フェーズで見直します。

```typescript
// 候補例。実装開始後、tsconfigなどで正式に決める。
import { calculateScore } from "@/game/scoring";
import { ResultPanel } from "./ResultPanel";
```

## Code Organization Principles

- 画面遷移は、開始前、カメラ準備中、カウントダウン、プレイ中、結果、エラーのように
  明確な状態として扱う。
- 壁パターンは、描画情報と判定情報を混同しすぎない。
- 衝突判定は、調整値をデータとして持てるようにし、コード変更なしで難易度を調整しやすくする。
- カメラや姿勢検出が使えない環境でも、可能な範囲でエラー表示やフォールバック検討ができる構成にする。
- 参考リポジトリのディレクトリやファイル名をそのまま再現しない。

## Documentation Rules

- READMEには、実装が入った時点でセットアップ、起動方法、カメラ利用条件を書く。
- `.kiro/specs/` には、機能ごとの要求、設計、タスクを置く。
- `.kiro/steering/` は、ファイル一覧ではなく判断基準を書く。
