# Design Document

## Overview

`arcade-sports-ui-refresh`は、既存のゲーム状態とロジックを維持したまま、
タイトルから結果までの表示を「近未来アーケード × 体感スポーツ」へ統一する。

UIはCanvasゲームを主役とし、画面外周の情報表示、短い状態演出、明確な操作階層で
ゲームらしさを作る。装飾用カードを増やさず、色、線、タイポグラフィ、余白、動きの
共通ルールによって一貫性を持たせる。

## Goals

- タイトル画面でゲーム内容と開始操作を即座に理解できるようにする。
- 準備画面で位置合わせ状態を文章を読み込まず判断できるようにする。
- プレイ画面で壁、安全領域、アバター、HUDを明確に分離する。
- 成功、失敗、速度上昇、結果を短く強い視覚反応で伝える。
- デスクトップ、モバイル、低い画面で主要操作と情報を画面内に維持する。
- 動きを抑える設定とキーボード操作を維持する。

## Non-Goals

- ゲームルール、壁データ、衝突判定、スコア、速度の変更。
- 姿勢検出、カメラ、キャリブレーション判定の変更。
- 効果音、BGM、ランキング、ユーザー管理の追加。
- 3D描画や動画素材の導入。

## Existing Architecture

UI刷新は既存の責務境界を維持する。

- `src/App.tsx`: 画面状態と画面コンポーネントの切り替え。
- `src/components/`: タイトル、準備、プレイ、結果、エラーなどの表示。
- `src/rendering/canvasRenderer.ts`: 背景、壁、安全領域、アバターのCanvas描画。
- `src/style.css`: 全画面のレイアウトと視覚スタイル。
- `src/game/`: ゲーム状態、進行、判定、スコア。今回の変更対象外を基本とする。

UIコンポーネントは表示と操作イベントだけを扱い、演出のためにゲーム結果を再計算しない。

## Visual System

### Color Roles

CSSカスタムプロパティで色の役割を定義する。画面ごとの直接的な色指定を減らし、
状態と色の対応を一貫させる。

| Role | Value | Usage |
|---|---|---|
| `--surface-0` | `#080b0d` | Canvas外背景、最暗部 |
| `--surface-1` | `#11171a` | 通常画面背景 |
| `--surface-2` | `#192126` | 操作面、HUD背景 |
| `--text-primary` | `#f7faf9` | 見出し、主要数値 |
| `--text-secondary` | `#aebbc0` | 補助文、ラベル |
| `--safe` | `#46f0c4` | 安全領域、準備完了、成功 |
| `--action` | `#20b8e6` | 主操作、フォーカス |
| `--score` | `#ffd447` | スコア、速度上昇 |
| `--danger` | `#ff5d68` | 壁、失敗、エラー |
| `--neutral-line` | `#34434a` | 境界線、未完了状態 |

安全領域と壁はCanvas内で最も彩度の高い要素とする。通常パネルは無彩色へ寄せ、
アクセント色が画面全体を占有しないようにする。

### Typography

- フォントは既存のシステムUIフォントを維持し、外部フォント読込を追加しない。
- タイトル画面のH1だけを大型表示とし、結果、準備、エラーは画面密度に合うサイズにする。
- 数値は`font-variant-numeric: tabular-nums`を使い、カウントやスコアで幅を固定する。
- 負のletter-spacingは使用しない。
- 英語のeyebrowは補助表示に留め、日本語の状態・操作を主表示にする。

### Shape and Depth

- 基本の角丸は8px、進捗バーは4px以下とする。
- 画面セクションを浮いたカードにせず、背景上の直接レイアウトとして構成する。
- HUD、繰り返し統計、状態確認など、境界が必要な小要素だけをパネル化する。
- 影は操作面の分離に限定し、発光は安全領域と短い判定演出へ限定する。

## Screen Design

### Title Screen

タイトル画面は左右分割カードではなく、Canvasで描いたゲーム場面を背景に持つ
一体型画面とする。

構成:

1. 背景全面に、壁、安全領域、ニュートラルアバターを使った低速デモを表示する。
2. 前景左下または中央下へゲーム名と短い説明を直接配置する。
3. 「ゲーム開始」を単一の主操作として配置する。
4. 背景と文字の間には局所的な暗色オーバーレイを置き、文字をカードへ入れない。
5. 第一ビューポート下端に次の情報が少し見える高さを確保する。

デモは既存`renderGameCanvas`の入力を固定し、ゲーム状態やスコアを進行させない。
`requestAnimationFrame`で壁進行だけを往復または再開し、`prefers-reduced-motion`では
判定位置の静止画を描画する。

### Camera Preparation

カメラ映像を全面表示し、状態UIを画面端へ配置する既存構造を維持する。

- 左上: 画面名と短い状態。
- 右側: アバター選択、位置合わせ4項目、開始状態。
- 下部: カメラ開始、モック開始、タイトル復帰。
- 位置合わせ項目は記号、色、短いラベルを主とし、詳細文は十分な高さがある場合だけ表示する。
- 準備完了時は安全色の枠と進捗で自動開始待機を示す。
- モバイルでは状態パネルを下部シートとして配置し、カメラ映像上の人物中心を空ける。

### Countdown

- 背景は開始時と同じCanvas場面を使用する。
- 中央に固定寸法の数字領域を置き、`3`、`2`、`1`でレイアウトを動かさない。
- 数字切り替え時に短いscaleとopacityの変化を行う。
- 背景Canvasの壁は開始位置で待機させ、開始後の方向を予告する。
- タイトル復帰は画面端へ低い優先度で残す。

### Playing

Canvasを全画面表示し、HUDは上部一列を基本とする。

- 左からハート、スコア、ミス、速度を配置する。
- ハート領域は5個分の固定幅を持ち、省略しない。
- HUDの各領域は数値の桁数が変わっても寸法を維持する。
- 姿勢状態はHUDとは別に、正常時は小さく、未検出時だけ中央寄りへ強く表示する。
- 終了操作は右下に置き、主操作色を使わない。
- 判定フィードバックは下部中央へ短く表示し、Canvasの人物中心を覆わない。

### Result

- 最終スコアを画面上部の主要数値として表示する。
- クリア枚数、最高速度、ミス数は横一列、狭い画面では2列で表示する。
- 「もう一度プレイ」を主操作、「タイトルへ戻る」を副操作とする。
- 自動復帰は残り秒数と連続進捗バーで表示する。
- 統計はカードを入れ子にせず、区切り線と数値階層で整理する。

### Error

- エラー色は画面上端、状態記号、見出しへ限定し、背景全体を赤くしない。
- 問題の説明と対処を1つの短い本文へまとめる。
- 「再試行」を主操作、「タイトルへ戻る」を副操作とする。
- 自動復帰表示は結果画面と同じコンポーネントを使う。

## Canvas Rendering

### Background

現在の単純な斜めグラデーションを、奥行きが分かる暗色の競技空間へ変更する。

- 暗い単色を基準にする。
- 消失点へ向かう床線と側面線を低コントラストで表示する。
- 壁が近づくほど背景線を強くせず、壁と安全領域の視認性を優先する。
- 装飾用のオーブ、ぼかし円、過剰な粒子は追加しない。

### Wall and Safe Area

- 壁は`--danger`相当のコーラル色を基準とする。
- 安全領域は`--safe`相当の色で、半透明塗りと太い輪郭を持つ。
- 壁が遠い場合も安全領域の輪郭が消えない最小線幅を設定する。
- 判定成功時は安全領域の発光を短時間強める。
- 判定失敗時は壁輪郭と画面外周を短時間危険色で反応させる。

描画層へ渡す判定演出状態は表示用の値とし、判定結果の決定権を持たせない。

### Avatar

- 既存3種類のアバターを維持する。
- 壁、安全領域、背景のどれとも同化しないアウトラインを追加する。
- 判定に使う領域表示は常時強く見せず、必要な調整時または判定直前に識別できる濃度とする。

## Component Design

### New Components

#### `TitleGamePreview`

責務:

- タイトル背景のCanvasサイズを監視する。
- 固定モック姿勢と壁パターンを`renderGameCanvas`へ渡す。
- 表示専用の壁進行を管理する。
- reduced motionでは静止描画にする。

#### `GameStatusHud`

責務:

- ハート、スコア、ミス、速度を安定寸法で表示する。
- 数値や状態を再計算せず、propsを表示する。

#### `JudgmentOverlay`

責務:

- 成功、失敗、判定不能、速度上昇の短い表示を行う。
- 状態ごとの文言と視覚クラスを選ぶ。

既存`AutoReturnCountdown`は継続利用し、結果とエラーで共通化する。

### Existing Component Changes

- `TitleScreen`: Canvasプレビューを含む全画面構成へ変更。
- `CalibrationPanel`: 状態項目を短い視覚表現へ整理。
- `GameScreen`: HUDと判定表示を子コンポーネントへ分離。
- `ResultScreen`: 最終スコアを第一階層へ変更。
- `ErrorScreen`: エラー状態、対処、操作の階層を整理。
- `App`: 既存状態とイベントを新しい表示コンポーネントへ渡す。状態遷移は変更しない。

## Motion Design

| Event | Duration | Motion |
|---|---:|---|
| Button hover/focus | 120ms | 色と1px以内の位置変化 |
| Countdown number | 450ms以内 | scaleとopacity |
| Success | 500ms以内 | 安全領域の発光、スコア表示 |
| Miss | 350ms以内 | 外周フラッシュ、ハート反応 |
| Speed up | 700ms以内 | スコア色の短い強調 |
| Auto return bar | 全待機時間 | 実時間に同期した連続減少 |

ループする装飾アニメーションはタイトルの壁デモだけとする。プレイ中はゲーム進行以外の
常時アニメーションを追加しない。

`prefers-reduced-motion: reduce`では以下を適用する。

- タイトルデモを静止させる。
- scale、移動、フラッシュを無効化する。
- 判定結果は色、輪郭、文言の即時切り替えだけで示す。
- 自動復帰バーは滑らかな幅変化を維持できるが、装飾的な脈動は行わない。

## Responsive Design

### Wide Desktop: 1200px以上

- タイトル文字は背景Canvasへ直接重ねる。
- 準備画面は右側固定パネルを使う。
- HUDは上部一列で表示する。
- 結果統計は横一列とする。

### Tablet and Narrow Desktop: 641pxから1199px

- タイトルの文字幅を制限し、開始操作を画面下寄りへ配置する。
- 準備パネルを画面下部へ移動できるようにする。
- HUDは一列を維持し、ラベルと数値を縮小する。
- 結果統計は2列へ変更する。

### Mobile: 640px以下

- タイトルのH1をモバイル用の固定段階へ下げる。
- 主操作と副操作を原則全幅にする。
- HUDは4列を維持し、ハート列だけ広く確保する。
- 姿勢警告はHUD下へ移動する。
- 結果統計は2列、非常に狭い場合は1列にする。
- safe-area-insetをボタンとHUDへ適用する。

### Low Height: 560px以下

- 補助文と詳細説明を省略する。
- 準備パネルは横方向に再配置する。
- カウントダウン数字と結果余白を縮小する。
- 主要操作、状態、Canvasを画面内に残す。

## Accessibility

- 既存の見出し構造とARIAラベルを維持する。
- 数値の視覚更新で読み上げを毎フレーム発生させない。
- 判定結果は`aria-live`で短い文言だけを通知する。
- フォーカスリングは3pxの`--action`または白で表示する。
- 色だけに依存せず、文言、記号、輪郭の組み合わせを使う。
- ボタンの最小高さは44px以上とする。

## Testing Strategy

### Unit and Type Validation

- `npm run typecheck`
- `npm test`
- `npm run build`

UIコンポーネントへ純粋な表示変換関数を追加した場合は、その関数をユニットテストする。
ゲームロジックのテスト期待値は変更しない。

### E2E

既存モック姿勢E2Eで以下を維持する。

- タイトルから準備へ進める。
- モック姿勢でカウントダウンとプレイへ進める。
- 結果から再プレイまたはタイトルへ戻れる。
- 準備、カウントダウン、プレイから展示リセットできる。

### Visual Verification

Playwrightで以下のスクリーンショットを取得する。

- 1440×900: タイトル、準備、カウントダウン、プレイ、結果、エラー。
- 390×844: タイトル、準備、プレイ、結果。
- 960×540: 準備、プレイ、結果。

各画像で以下を確認する。

- Canvasが空白でない。
- 壁、安全領域、アバターが画面内にある。
- HUD、ボタン、見出し、状態文が重ならない。
- ハート5個と主要数値が省略されない。
- 最長の日本語文言がコンテナからはみ出さない。

## Risks and Mitigations

### Canvasデモによる負荷

タイトルでCanvasを常時更新すると低性能端末の負荷が増える。

対策:

- 1枚の壁と固定モック姿勢だけを描画する。
- タブが非表示のときはブラウザの`requestAnimationFrame`停止に従う。
- reduced motionでは静止描画にする。

### 演出とゲーム判定の混同

表示アニメーションが判定時刻やゲーム状態へ影響する可能性がある。

対策:

- 演出状態は描画層とUI層だけで扱う。
- 既存の`JudgmentResult`を入力にし、判定を再計算しない。
- ゲームループのテスト期待値を変更しない。

### 狭い画面でのHUD圧迫

ハート5個と3つの数値を一列にすると、狭い画面で不足する可能性がある。

対策:

- ハート列へ固定最小幅を割り当てる。
- ラベルと数値に段階的なモバイルサイズを設定する。
- Playwrightで390px幅と最長数値を確認する。
